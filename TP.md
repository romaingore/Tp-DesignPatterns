# 🧪 TP Individuel – Comprendre et implémenter des Design Patterns

## 🎯 Objectif pédagogique

L’objectif de ce TP est de vérifier que vous **comprenez réellement** les Design Patterns, et que vous êtes capables :

- D’en expliquer le fonctionnement
- D’identifier leur problématique
- De les implémenter vous-même
- D’inventer un exemple original
- De produire du code structuré et cohérent

⚠️ **Attention : vous ne devez documenter QUE les Design Patterns que vous avez compris.**  
Si vous ne comprenez pas un pattern, vous ne passez pas au suivant.

---

# 📚 Ressource obligatoire

Vous devez vous baser exclusivement sur le site :

👉 https://refactoring.guru/fr/design-patterns

⚠️ Règles strictes :

- ❌ Interdiction de copier les exemples du site  
- ❌ Interdiction de reprendre les mêmes contextes (canard, éditeur graphique, transport, etc.)  
- ❌ Interdiction de traduire un exemple existant  
- ✅ Obligation d’inventer un contexte métier totalement différent

# 📌 Pour CHAQUE Design Pattern compris

Vous devez produire :

---

## 1️⃣ Une explication rédigée (fichier explication.md)

Votre explication doit contenir :

- 📌 Nom du pattern  
- 🎯 Problème qu’il résout  
- 🧠 Principe de fonctionnement  
- 🏗 Structure (rôles des classes)  
- 📈 Avantages  
- ⚠️ Inconvénients  
- 🧩 Cas d’usage réel possible  

⚠️ Vous devez reformuler avec vos propres mots.

---

## 2️⃣ Une mise en pratique OBLIGATOIRE

Vous devez fournir :

- Un exemple inventé  
- Un contexte métier original  
- Un code fonctionnel  
- Plusieurs fichiers si nécessaire  
- Un petit script de démonstration (main, index, etc.)  

---

# 🛠 Langage autorisé

Vous pouvez choisir librement :

- PHP  
- Java  
- Python  
- C#  
- JavaScript / TypeScript  
- Autre langage orienté objet  

⚠️ Le langage doit supporter la POO.

---

# 📌 Nombre minimum de patterns

Vous devez en traiter **au minimum 4**, parmi :

## 🔹 Création
- **Singleton**  
- **Factory Method**  
- Abstract Factory  
- _Builder_

## 🔹 Structure
- _Adapter_  
- **Decorator**  
- _Facade_
- Composite  

## 🔹 Comportement
- _Strategy_
- **Observer**
- Command
- State

⚠️ la liste n'est pas exhaustive !!!

⚠️ Vous ne devez traiter que ceux que vous avez compris.

---

# 📊 Contraintes importantes

## ❗ Interdictions

- Copier-coller du site  
- Reprendre les mêmes exemples  
- Utiliser un générateur automatique  
- Rendre un code non testé  
- Faire une explication vague  

---

## ✅ Obligations

- Inventer un scénario original  
- Justifier pourquoi le pattern est pertinent  
- Respecter les principes SOLID  
- Produire un code propre et structuré  
- Mettre des commentaires explicatifs  

---

# 📦 Rendu attendu

- Un lien vers un repo github.
- Code exécutable  
- Explications en Markdown ou PDF  
- README principal expliquant :  
    - Langage utilisé  
    - Comment exécuter les exemples  

---

# 📝 Évaluation

## Compréhension (40%)
- Explication claire  
- Bonne identification du problème  

## Qualité du code (30%)
- Lisibilité  
- Structure  
- Respect du pattern  

## Pertinence de l’exemple (20%)
- Originalité  
- Cohérence métier  

## Rigueur (10%)
- Respect des consignes  
- Pas de copie  
- Code fonctionnel  

---

# 🎓 Objectif réel du TP

Ce TP ne vise pas à produire beaucoup.  
Il vise à produire **juste ce que vous comprenez vraiment**.

Un étudiant qui rend 4 patterns parfaitement compris aura une meilleure note qu’un étudiant qui en rend 10 mal compris.
